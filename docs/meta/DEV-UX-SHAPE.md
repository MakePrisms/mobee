# Dev lane — buyer UX shape (config · wallet · profile)

Happy-path-first shape for the `dev` exploratory lane (cut off main @ `0145ed3`).
Clean code we later rip pieces from, rethink, and back-pull to main — not a throwaway
spike. The deep money bar returns at back-pull; on `dev` it is teammate eyes + evidence.
**Testnut only. No real funds. The key never leaves the box.**

This shapes the seamless buyer UX (piece-8). It does not lock it — name a better shape
when you hit one. "Keep it simple for now, but build to where we want to go."

## One binary · one core · two surfaces

- **`mobee-core`** (exists): payment SM (piece-6) + delivery-verify (piece-7) + gateway.
  Surfaces stay thin; all money / verify logic lives in core.
- **One `mobee` binary**, two surfaces over the same core:
  - `mobee mcp` — MCP server over stdio. **Newline-delimited JSON-RPC** (NOT
    `Content-Length` framing — the spike's LSP framing hung Claude Code's MCP client).
    Read may accept both framings; writes are newline-delimited; diagnostics to stderr only.
  - `mobee …` — human CLI, same core calls.
- **Zero-arg onboarding:** `claude mcp add mobee -- mobee mcp`. First run bootstraps
  `~/.mobee` with working defaults — no flags, no pasted secrets.

## `~/.mobee` — packaged config

- `config.toml` — relay URL (default: mobee-relay), mint URL (default: **testnut**),
  budget caps (per-job + total). Every field has a working default.
- key — autogenerated on first run, mode `0600`, **never printed / logged / committed**.
- `wallet/` — cdk wallet store (testnut).

## Wallet — allowance-not-bank

The wallet holds funds; the MCP is the *spend authority* the agent operates under. That
gap is the whole safety story of handing an autonomous agent money.

- **`setup_wallet` tool** — autogen key if absent → create the `~/.mobee` wallet → return a
  Lightning fund invoice (cdk mint quote). The key never appears in the response.
- **Fund** — user pays the LN invoice → cdk mints tokens into the wallet.
- **Allowance gate** — the MCP enforces budget caps *before* a pay reaches the piece-6 SM:
  - per-job cap + total (rolling / session) cap, from config;
  - a spend over a cap is refused at the MCP layer — the SM never sees it;
  - this is a security control, not just UX: it bounds worst-case loss.

## Profile — people, not hex

## Happy path (the acceptance target)

fresh buyer → `claude mcp add` → `setup_wallet` (key + wallet + fund invoice) → pay the
invoice (testnut) → post a job → seller delivers → buyer verifies the work → pay within
allowance → paid result. Documented steps only; the transcript is the evidence.

## Security seams — named here, teeth at back-pull to main

Shaped in `dev`, hardened under the money bar at back-pull — **except the transport
allowlist**, which must land the moment a real git fetch is wired (dev exercises one).

- **Transport-scheme allowlist (HARD; lands with the delivery-verify caller, not deferred):**
  before the buyer MCP calls delivery-verify with a *seller-supplied* git locator, allowlist
  `https` + relay-git and **refuse `ext::` / `file` / `ssh`**. `ext::` = arbitrary command
  execution (piece-7's carried RCE). Even in dev this is load-bearing the instant there is a
  real fetch — it is a few lines; do not ship the caller without it.
- **Independent-commitment bind:** the buyer's `delivery_integrity_hash` must be the buyer's
  *own* commitment to the deliverable (from its content check), never a blind copy of the
  seller-advertised oid — else the bind is circular. `authorize_pay` builds the key from the
  buyer's verified content decision.
- **Bound-parser caller contract:** use `parse_bound_git_delivery` (offer-bound repo+branch),
  never the unbound parser.
- **Key-paste designed out:** onboarding autogens + stores the key locally; nothing ever asks
  a human to paste a key (the leaked-key scar — the UX itself prevents it).
- **Standing:** testnut only; no secrets in channel / logs / commits.
